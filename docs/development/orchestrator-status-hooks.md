# Orchestrator Status Hooks

Issue #40 introduces **daemon-owned orchestrator status** plus a user-level
hook surface so display sinks like tmux, desktop notifications, or webhooks
all read the same authoritative aggregate state instead of racing with each
other through inherited Claude hooks.

## Mental model

- The Claude Code supervisor is the cockpit.
- The `agent-orchestrator` daemon is the control plane: it owns durable run
  state, worker lifecycle, notifications, and the aggregate orchestrator
  status.
- User-level status hooks are display sinks. Tmux is one optional sink; the
  daemon ships zero tmux logic.

The supervisor signals turn boundaries via Claude Code lifecycle hooks
generated by the launcher (`UserPromptSubmit`, `Notification`, `Stop`,
`SessionStart`, `SessionEnd`). Worker runs contribute to the same aggregate
status because the daemon stamps `metadata.orchestrator_id` on every
`start_run` / `send_followup` whose harness pinned an orchestrator id.

## Aggregate states

Five states drive what user hooks render:

| State | Meaning |
|---|---|
| `in_progress` | Supervisor turn active **or** any owned worker run is running. Running children dominate `waiting_for_user` while children are alive (AC8 invariant). |
| `waiting_for_user` | Supervisor needs user input or approval and no owned runs are running. |
| `idle` | Supervisor is inactive and no owned worker runs are running. |
| `attention` | An owned worker run failed or needs acknowledgement. |
| `stale` | Supervisor reported `SessionEnd` or has not heart-beated within `STALE_AFTER_SECONDS` (= 600) and the owned-run set is empty. |

The 5-rule live-state precedence is encoded in
`src/daemon/orchestratorStatus.ts` (`computeOrchestratorStatusSnapshot`).

## v1 hook payload

User hooks read JSON on stdin. v1 schema:

```json
{
  "version": 1,
  "event": "orchestrator_status_changed",
  "event_id": "01J...",
  "previous_status": "idle",
  "emitted_at": "2026-05-06T12:00:00.000Z",
  "orchestrator": {
    "id": "01J...",
    "client": "claude",
    "label": "payments-refactor",
    "cwd": "/path/to/worktree"
  },
  "status": {
    "state": "in_progress",
    "supervisor_turn_active": false,
    "waiting_for_user": false,
    "running_child_count": 2,
    "failed_unacked_count": 0
  },
  "display": {
    "tmux_pane": "%12",
    "tmux_window_id": "@4",
    "base_title": "payments-refactor",
    "host": "host"
  }
}
```

## hooks.json v1 schema

User-level config lives at `~/.config/agent-orchestrator/hooks.json` on
Linux **and** macOS. The schema is **closed at every level**: any unknown
key is rejected so a future v2 can add fields additively without breaking
v1 consumers.

```json
{
  "version": 1,
  "hooks": {
    "orchestrator_status_changed": [{
      "type": "command",
      "command": "~/.config/agent-orchestrator/hooks/tmux-status.sh",
      "timeout_ms": 1500
    }]
  }
}
```

This is the same Claude-parity shell-string shape used by
`~/.claude/settings.json` `hooks` entries. The daemon executes `command` via
`child_process.spawn(command, { shell: true })` and pipes the JSON payload
on stdin. Stdout / stderr are captured to
`<store_root>/hooks/<orchestrator_id>/<event>-<event_id>.log` (mode 0o600).

### Threat model

`~/.config/agent-orchestrator/hooks.json` is **user-owned and user-trusted**,
exactly like `~/.claude/settings.json`. The daemon **never** interpolates
model-supplied, worker-supplied, or run-supplied data into `command`; the
JSON payload reaches the script via stdin only. There is no remote-write
path to this file; user-side write protection (file mode, `chown`) is your
responsibility.

If you reference secrets, prefer environment variables in the hook entry's
`env` block over inlining them in the `command` shell string.

### Execution semantics

- Best-effort: the daemon never awaits hook completion before continuing
  orchestration.
- Per-entry `timeout_ms` defaults to 1500 ms and is capped at 5000 ms;
  timeouts SIGKILL the spawned shell (and its process group on POSIX).
- Restricted env passthrough: `AGENT_ORCHESTRATOR_ORCH_ID`,
  `AGENT_ORCHESTRATOR_EVENT`, `PATH`, `HOME`, `LANG`, plus the entry's own
  `env` overrides. Other variables are dropped.
- Per-orchestrator hook execution counters (`emitted`, `successes`, `failures`,
  `timeouts`, `log_capture_failed`) are tracked on the in-process executor
  for diagnostics. Surfacing them through `get_observability_snapshot` is
  deferred to a follow-up; today the counters are accessible programmatically
  via the daemon's `OrchestratorHookExecutor.counters` field.

## Reading status from a shell script

Use the `agent-orchestrator supervisor status` CLI subcommand to read the
current orchestrator status without subscribing to hooks. It returns JSON
on stdout:

```bash
agent-orchestrator supervisor status --orchestrator-id "$AGENT_ORCHESTRATOR_ORCH_ID"
```

The top-level `agent-orchestrator status` subcommand is reserved for
**daemon** status and is not overloaded with orchestrator-status semantics.

## Worker isolation

Workers are isolated from supervisor display surfaces:

- The daemon strips `TMUX`, `TMUX_PANE`, `STY`, `WEZTERM_PANE`,
  `KITTY_WINDOW_ID`, `ITERM_SESSION_ID`, and `WT_SESSION` from the worker
  subprocess env, and adds `AGENT_ORCHESTRATOR_WORKER=1`,
  `AGENT_ORCHESTRATOR_WORKER_RUN_ID`, and
  `AGENT_ORCHESTRATOR_PARENT_ORCHESTRATOR_ID`. User hook scripts can use
  these to no-op when running under a worker.
- Claude worker runs receive a generated `settings.json` with
  `disableAllHooks: true`, so user `~/.claude/settings.json` hooks do not
  fire under workers.

## Remote Control opt-in

`agent-orchestrator claude --remote-control` embeds the documented
`remoteControlAtStartup` and `agentPushNotifEnabled` keys in the generated
supervisor settings (Decision 12). `--remote-control-session-name-prefix
<prefix>` is forwarded to Claude as a CLI flag. Default is off. Remote
Control enablement is sensitive to the local Claude version; T14's offline
config smoke records the version observed by `--print-discovery`.

## Example: tmux pane rename

`examples/hooks/tmux-status.sh` renames the pane the supervisor was
launched in. ASCII state labels are the default (`[in_progress]`,
`[waiting]`, `[attention]`, `[stale]`, `[idle]`). Set
`AGENT_ORCHESTRATOR_HOOK_USE_EMOJI=1` in the hook entry's `env` block to
opt into emoji.
