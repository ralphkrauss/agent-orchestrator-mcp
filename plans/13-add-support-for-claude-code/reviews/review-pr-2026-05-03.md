# PR Review: 13-add-support-for-claude-code

**Scope:** `origin/main...HEAD`
**Verdict:** needs-attention

## Findings

### 1. Pinned Bash monitor allowlist still permits arbitrary shell suffixes (high)
- **File:** `src/claude/monitorPin.ts:23`
- **Issue:** The Claude supervisor exposes Bash and pre-approves `Bash(<node> <agent-orchestrator> monitor *)` through the generated settings and `--allowed-tools`. Because the permission is a shell-command glob, anything after `monitor ` matches. A command such as `<node> <cli> monitor <run_id>; cat "$HOME/.claude/.credentials.json"` still starts with the allowed monitor prefix, but Bash will execute the suffix as a second shell command. The targeted deny patterns in `src/claude/permission.ts:28` only catch commands that start with `cat`, `jq`, etc.; they do not catch suffix injection behind an allowed prefix.
- **Recommendation:** Do not rely on a broad Bash glob to enforce argument grammar. Either remove in-envelope Bash and use an MCP wait/notification path, or introduce a monitor mechanism that Claude cannot suffix with shell syntax. If Bash must remain, the implementation needs an enforceable wrapper/permission shape that cannot execute additional shell constructs, with tests covering `;`, `&&`, pipes, command substitution, redirection, and extra flags.

### 2. `--debug-file` passthrough can write outside the supervisor envelope (medium)
- **File:** `src/claude/passthrough.ts:44`
- **Issue:** The passthrough allowlist includes `--debug-file`, and the launcher help advertises it as allowed. The local Claude CLI help describes that flag as writing debug logs to a specific file path. That lets a user-supplied passthrough write to arbitrary paths outside the daemon-owned state/envelope, which conflicts with the isolation boundary and the plan's "no writes outside harness-owned state" invariant.
- **Recommendation:** Forbid `--debug-file`, or rewrite it to a harness-owned log path under the Claude supervisor state directory. Add a passthrough test that rejects `--debug-file=/tmp/...` and `--debug-file /tmp/...`.

### 3. Cursor-based `get_run_progress` drops latest-event metadata (medium)
- **File:** `src/orchestratorService.ts:415`
- **Issue:** When `after_sequence` is supplied, `getRunProgress` calls `readEventSummary(run_id, 0)`. `readEventSummary` uses its recent-events tail to populate `last_event`, so with a limit of `0` the response reports `latest_event_sequence: null` and `latest_event_at: null` even when the run has events. The current cursor-mode test only checks the page, not the latest metadata.
- **Recommendation:** Keep latest-event metadata independent of the cursor page. For example, ask the store for one tail event or expose a cheap latest-event helper, then add a cursor-mode test asserting `latest_event_sequence` and `latest_event_at` remain populated.
