# Review: Uncommitted Changes

Date: 2026-05-04
Scope: `git diff HEAD`

## Findings

### P1: Claude Bash is exposed through a denylist, so write-capable commands still slip through

The launcher exposes the `Bash` tool with `--tools Read,Glob,Grep,Bash,Skill`,
while the settings only deny selected shell patterns and selected mutating git
subcommands. The intended contract is read-only Bash/git plus the pinned monitor,
but this denylist does not cover common write paths such as `node -e
"require('fs').writeFileSync(...)"`, `python -c "open(..., 'w')"`, `bash -c
"touch /tmp/x"`, `git rm`, `git fetch`, or package-manager commands. Because
bare `Bash` is not denied, any command not matching the list depends on model
prompt compliance rather than an enforceable permission boundary.

References:

- `src/claude/launcher.ts:397`
- `src/claude/permission.ts:29`
- `src/claude/permission.ts:123`

### P1: `upsert_worker_profile` can create or overwrite arbitrary profile-shaped files

The new MCP tool accepts arbitrary `profiles_file` and `cwd`, resolves that path,
and writes a manifest there. If the file is missing, it starts from an empty
manifest and creates the parent directory plus the file. Since this tool is
allowlisted for the Claude supervisor, it becomes a write primitive even though
the supervisor is documented as unable to directly write workspace files except
for profile repair. At minimum the supervisor path should be constrained to the
launcher-provided writable profiles manifest, or the tool needs an explicit
server-side policy gate for arbitrary paths.

References:

- `src/contract.ts:374`
- `src/orchestratorService.ts:323`
- `src/orchestratorService.ts:361`

### P2: `just local ...` interpolates raw args into the shell

The local passthrough recipes use `set -- {{args}}`, which does not preserve CLI
arguments safely. A dry run shows shell metacharacters become live shell syntax:
`just --dry-run local 'status; echo SHOULD_NOT_RUN'` renders `set -- status;
echo SHOULD_NOT_RUN; ...`. This breaks the intended "same CLI experience" for
quoted prompt text and can execute host-shell commands before the local CLI runs.

References:

- `justfile:54`
- `justfile:58`

### P2: The branch plan still records the superseded Claude behavior as current

The implementation and user-requested behavior now rely on target-workspace
launches, slash commands, mirrored skills, and the pinned Bash monitor. The
branch plan still says the current implementation excludes Bash, waits through
`wait_for_any_run`, runs from the stable envelope, and exposes only
orchestrate-* skills. Since these plan files are loaded by orchestration skills
as source context, leaving the old behavior marked current increases the chance
that future orchestrated runs will "fix" the implementation back to the old
shape.

## Resolution (2026-05-04)

- P1 Bash deny list extended in `src/claude/permission.ts` to cover script
  interpreters, inline-script flags (`-e`, `-c`, `--eval`, `--exec`), package
  managers, network/file-transfer tools, and additional mutating/network git
  subcommands. Coverage asserted in `src/__tests__/claudeHarness.test.ts`.
- P1 `upsert_worker_profile` constrained to the harness-provided manifest path:
  `src/claude/config.ts` now sets
  `AGENT_ORCHESTRATOR_WRITABLE_PROFILES_FILE` on the MCP server entry, and
  `src/serverPolicy.ts` (used from `src/server.ts`) rejects requests whose
  resolved `profiles_file` does not match the pin. Behavior covered by
  `src/__tests__/serverPolicy.test.ts` and the existing harness tests.
- P2 `justfile` recipes now use `set positional-arguments`; the local /
  agent-orchestrator recipes pass `"$@"` instead of expanding `set --
  {{args}}`, so `just local 'status; echo SHOULD_NOT_RUN'` now passes the
  whole literal string as a single CLI argument and does not execute host
  shell commands.
- P2 `plans/13-add-support-for-claude-code/plans/13-claude-code-support.md`
  current-implementation note, deliverables list, and leak-proof test list
  updated to describe the target-workspace launch model, the
  `Read,Glob,Grep,Bash,Skill` built-in surface with the pinned monitor
  allowlist, comprehensive Bash deny list, the redirected user skill mirror,
  and the writable-profiles pin. The pinned Bash monitor is documented as
  the primary wake path; `wait_for_any_run` / `wait_for_run` are documented
  as denied in the Claude supervisor.
- `pnpm test` (full suite) ran clean: 238 passed, 1 skipped, 0 failed.
